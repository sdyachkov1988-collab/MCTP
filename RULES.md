# MCTP — Правила работы

## Контекст проекта
- Контекст и текущее состояние: `MCTP_context.md`
- Роадмап и контракты: `MCTP_Roadmap_v2-2.docx`
- Код проекта: zip архив или GitHub репо
- Инструкции для агентов: `AGENTS.md`

## Участники работы
- **Claude (чат)** — архитектурные решения, аудит, ТЗ, roadmap compliance
- **Claude Code** — реализация, рефакторинг, запуск тестов, работа с Git
- **Codex** — реализация по ТЗ
- **Claude (чат) выбирает** кому дать задачу — Claude Code или Codex — в зависимости от сложности

## Схема работы
1. Claude (чат) изучает контекст и текущее состояние репо
2. Claude (чат) даёт ТЗ на следующую версию
3. ТЗ отдаётся исполнителю — Claude Code или Codex
4. Исполнитель пишет код и прогоняет тесты
5. Результат (zip или файлы) скидывается Claude (чат) на аудит
6. Claude (чат) проводит аудит и даёт следующее ТЗ или список правок
7. После принятия версии — обновить `MCTP_context.md` и `AGENTS.md`
8. Коммит только после зелёных тестов

## Правила для ТЗ
- Только описание задачи, правила и Done criteria
- Никакого готового кода в ТЗ — исполнитель пишет сам
- Каждое ТЗ содержит:
  - Цель и scope
  - Anti-scope (что явно не делать)
  - Protected files (что нельзя трогать)
  - Done criteria (список условий приёмки)
  - Технические инварианты

## Технические инварианты (нарушение недопустимо)
- Все финансовые значения — только `Decimal`
- Все временные метки — только UTC-aware `datetime`
- `Symbol` — только типизированный объект, не plain string
- Константы — только из `mctp/core/constants.py`
- Strategy layer остаётся read-only
- Backtest использует только закрытые свечи
- Exchange-specific логика остаётся внутри adapter/runtime boundary
- Paper mode и testnet mode остаются разделёнными

## Protected scope (без явного разрешения не трогать)
- `mctp/runtime/testnet.py`
- `mctp/runtime/testnet_recovery.py`
- `mctp/runtime/testnet_streams.py`
- `mctp/runtime/testnet_exchange_boundary.py`
- `mctp/adapters/binance_spot_testnet_v1.py`
- `storage/` — accounting/reconciliation path
- `docs/v1_7_*` — operator artifacts

## Anti-scope (не реализовывать пока не дойдём по roadmap)
- multi-pair (только после v2.2 и выполнения критериев)
- futures (только после контракта 44)
- regime / anomaly / on-chain / ML
- allocation engine
- широкий рефакторинг без прямой необходимости

## Порядок аудита
1. Запустить тесты: `python -m pytest tests/ -q`
2. Проверить что все тесты зелёные
3. Сверить код с контрактами из roadmap
4. Проверить технические инварианты
5. Найти отклонения от ТЗ (scope creep, упрощения, пропуски)
6. Дать вердикт: принято / нужны правки / откат

## Правила коммитов
- Коммитить только зелёные тесты
- Формат: `vX.Y: краткое описание — N tests passing`
- Тегировать каждую принятую версию: `git tag vX.Y-final`
- Перед новой версией сохранять точку отката

## Как начать новый чат
Скинуть три файла:
1. `MCTP_context.md` — текущее состояние
2. `RULES.md` — этот файл
3. `MCTP_Roadmap_v2-2.docx` — контракты и roadmap

Этого достаточно чтобы Claude восстановил полный контекст.
Zip архива для аудита не нужен пока не начинается новая задача.
